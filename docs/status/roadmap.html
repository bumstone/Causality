2 어디까지 닫혔는가

완료된 기반과 아직 보장하지 않는 영역을 구분해 다음 구현 우선순위를 확인한다.

3 다음에 뭘 봐야 하나

남은 로드맵은 구현 공백이 아니라 “아직 보장한다고 말하면 안 되는 것”을 정리한다. 새 작업은 여기서 시작한다.

60초 프로젝트 설명

목적 에이전트 실행을 계약과 증거로 묶는다

Causality는 에이전트가 임의로 작업하지 않도록 목표 계약, 허용 도구, non-goal, stop 조건, verifier, HITL gate, append-only ledger를 한 흐름으로 묶는 로컬 우선 제어 하네스다.

핵심 가치 “했다”가 아니라 “증거가 있다”

완료 주장은 ledger evidence, verifier decision, gate decision으로 검증된다. PR #25 이후 verifier pass도 공허한 이름 2개가 아니라 rationale 또는 evidence ref가 있는 실질 pass만 quorum에 반영된다.

현재 표현
“전체 완성”이 아니라 “코어 제어 하네스는 기능적으로 닫힘”. 남은 로드맵은 제품화/운영 품질/시나리오 증거로 분리한다.

완료된 기반

완료 실행 게이트

run_task가 plan gate와 action gate를 강제한다. 도구, non-goal, 위험 action은 ExecutionAdapter를 지나며, 파일 read/write와 subprocess 호출은 ToolAdapter가 계약과 증거 기록을 묶는다.

완료 실패 환류

scoped failure는 HITL 승인 후 다음 계약의 non-goal로 들어가며, TTL과 active filter로 영구 ratchet을 피한다.

완료 스킬 재사용

SkillStore.recall이 promoted skill을 objective 관련도로 회수하고, authored skill을 우선한다.

완료 P2 스킬 품질

PR #25로 문자열 trace 수준을 넘어 tool, args, artifacts, outcome을 담는 structured step과 legacy step 호환이 들어갔다.

완료 파일/서브프로세스 어댑터

PR #29로 ToolAdapter가 run, write_text, read_text를 제공한다. allowed tool, non-goal, risk, write scope를 통과한 호출만 실행하고 tool/evidence 이벤트를 남긴다.

완료 L1 playbook vendoring

L1 _ROUTING의 번들 라벨이 더 이상 빈 문자열이 아니라 causality.playbooks의 구조화된 Playbook(순서 있는 phase·step)으로 vendoring됐다. 라우팅된 모든 라벨이 실재 playbook으로 resolve되거나 raise하며, TaskRun.playbooks로 run에 노출된다.

완료 ledger scale path

대규모 ledger를 위해 rotate()가 archive segment를 만들고 chain continuity를 보존한다. .idx 사이드카는 event_count()와 events_page()가 전체 파싱 없이 archive를 다루게 한다.

완료 프로젝트 온보딩

/onboard는 새 세션의 repo map, 현재 작업, 구현 계획, 검증 리스크를 모은다. PR #35 installer는 이 기준선을 client-native MCP activation과 연결한다.

완료 보안 redaction

secret key 이름, nested dict/list, hyphenated sk-, opaque bearer/basic token을 skill distill 경로에서 방어한다.

완료 verifier substance

blank evidence ref와 공허한 pass가 completion quorum을 채우지 못하도록 counts_as_pass와 review 집계가 보강됐다.

완료 intent fail-safe

민감 작업이 키워드 미스 때문에 TRIVIAL로 빠지지 않도록 destructive/access/credential 신호를 governed route로 보낸다.

완료 ledger 내구성

hash chain, atomic/fsync/flock, torn-tail 처리, size-guarded read cache, mutable isolation이 반영됐다.

완료 리뷰/문서 예산

review batch budget, document budget, PR review watch/autofix 규칙이 프로젝트 운영 절차로 들어왔다.

남은 로드맵

구현 순서와 수용 기준은 Plan 001 및 연결된 상세 spec을 기준으로 한다.

항목 정직한 현재 상태 다음으로 증명할 것
01 · 설치 활성화 (완료) client-native MCP 설정, host 보존/adopt, venv interpreter 고정, handshake와 durable install report가 구현됐다. trust/approval은 우회하지 않고 pending으로 보고한다. Spec 001 수용 테스트 완료. 다음 구현은 02 검증 evidence다.
02 · 검증 evidence 현재 verification 문자열은 실제 명령 실행이 아니라 evidence kind 요구사항에 가깝다. Spec 002: requirement ID, argv, exit code, artifact hash를 completion에 결합.
03 · MCP task lifecycle MCP는 init/context/evidence/workflow 조회만 제공한다. 계약, action, verdict, completion, reflection은 Python API에만 있다. Spec 003: durable task state와 gated lifecycle tools.
04 · API/browser adapters 파일/서브프로세스 adapter만 run 안에서 검증됐다. API와 browser action은 같은 계약/증거 모델로 실행되지 않는다. Spec 004: scope-gated API/A11y actions와 E2E evidence.
05 · workflow/debug loop workflow/playbook phase는 지침이며, debug 분류·가설 실패 escalation·failure dedup이 엔진 lifecycle에 완전히 연결되지 않았다. Spec 005: persisted phase runner와 root-cause loop.
06 · resume/skill operations failure와 candidate skill은 로컬 기록되지만, MCP resume·outcome·promotion과 terminal idempotency가 없다. Spec 006: durable resume와 HITL skill operations.

문서로서의 적합성 판단

적합 프로젝트 설명용

첫 화면에서 Causality가 “agent workflow kit”가 아니라 구체적으로 무엇을 통제하는지 보이도록 계약, gate, ledger, skill recall을 한 문장과 5단계 지도로 배치했다.

적합 변경 추적용

완료된 기반과 남은 보장 공백을 분리하고, 최신 검증 기준은 단일 상태 원천으로 연결한다.

주의 완성 선언 문서로는 부적합

API/browser 실행, playbook phase 자동화, 운영 자동화 검증이 남아 있으므로 “프로젝트 완성”이라고 말하면 과장이다. 이 문서는 “코어 하네스는 닫혔고, 제품화/운영 품질이 남았다”는 판단에 맞춰 읽어야 한다.

개선 여지 자동 생성 가능성

앞으로는 PR/ledger/ADR에서 상태를 추출해 HTML을 생성하면 최신성 리스크가 줄어든다. 지금은 수동 상태판이므로 갱신 commit과 test count를 계속 명시해야 한다.
