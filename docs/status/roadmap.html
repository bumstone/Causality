60초 프로젝트 설명

목적 에이전트 실행을 계약과 증거로 묶는다

Causality는 에이전트가 임의로 작업하지 않도록 목표 계약, 허용 도구, non-goal, stop 조건, verifier, HITL gate, append-only ledger를 한 흐름으로 묶는 로컬 우선 제어 하네스다.

핵심 가치 “했다”가 아니라 “증거가 있다”

완료 주장은 ledger evidence, verifier decision, gate decision으로 검증된다. PR #25 이후 verifier pass도 공허한 이름 2개가 아니라 rationale 또는 evidence ref가 있는 실질 pass만 quorum에 반영된다.

현재 표현
“전체 완성”이 아니라 “코어 제어 하네스는 기능적으로 닫힘”. 남은 로드맵은 제품화/운영 품질/시나리오 증거로 분리한다.

완료된 기반

완료 실행 게이트

run_task가 plan gate와 action gate를 강제한다. 도구, non-goal, 위험 action은 ExecutionAdapter를 지나며, 파일 read/write와 subprocess 호출은 ToolAdapter가 계약과 증거 기록을 묶는다.

완료 실패 환류

scoped failure는 HITL 승인 후 다음 계약의 non-goal로 들어가며, TTL과 active filter로 영구 ratchet을 피한다.

완료 스킬 재사용

SkillStore.recall이 promoted skill을 objective 관련도로 회수하고, authored skill을 우선한다.

완료 P2 스킬 품질

PR #25로 문자열 trace 수준을 넘어 tool, args, artifacts, outcome을 담는 structured step과 legacy step 호환이 들어갔다.

완료 파일/서브프로세스 어댑터

PR #29로 ToolAdapter가 run, write_text, read_text를 제공한다. allowed tool, non-goal, risk, write scope를 통과한 호출만 실행하고 tool/evidence 이벤트를 남긴다.

완료 L1 playbook vendoring

L1 _ROUTING의 번들 라벨이 더 이상 빈 문자열이 아니라 causality.playbooks의 구조화된 Playbook(순서 있는 phase·step)으로 vendoring됐다. 라우팅된 모든 라벨이 실재 playbook으로 resolve되거나 raise하며, TaskRun.playbooks로 run에 노출된다.

완료 보안 redaction

secret key 이름, nested dict/list, hyphenated sk-, opaque bearer/basic token을 skill distill 경로에서 방어한다.

완료 verifier substance

blank evidence ref와 공허한 pass가 completion quorum을 채우지 못하도록 counts_as_pass와 review 집계가 보강됐다.

완료 intent fail-safe

민감 작업이 키워드 미스 때문에 TRIVIAL로 빠지지 않도록 destructive/access/credential 신호를 governed route로 보낸다.

완료 ledger 내구성

hash chain, atomic/fsync/flock, torn-tail 처리, size-guarded read cache, mutable isolation이 반영됐다.

완료 리뷰/문서 예산

review batch budget, document budget, PR review watch/autofix 규칙이 프로젝트 운영 절차로 들어왔다.

남은 로드맵

항목 정직한 현재 상태 다음으로 증명할 것
End-to-end 시나리오 PR #28로 대표 루프 테스트가 추가됐지만, 실패 종류와 tool adapter 경로를 넓힌 장기 시나리오 증거는 더 필요하다. 계약 생성 → 실행 gate → failure 환류 → skill distill/recall → tool adapter → verifier completion까지 이어지는 복수 시나리오 테스트.
실행 playbook 번들 라벨은 구조화된 playbook으로 vendoring됐고(phase·step), 파일/서브프로세스 어댑터도 들어왔다. 다만 phase는 에이전트가 따르는 지침이며 엔진이 자동 실행하지 않고, API/browser 실사용 어댑터는 아직 없다. API/browser 실행 어댑터와, playbook phase를 실제 실행/검증으로 연결하는 경로를 테스트로 고정.
운영 자동화 설정 PR watch/autofix 정책은 있으나 외부 App 설치, secret 설정, repository rule과 결합되어야 완전한 자동 운영이 된다. 설치 상태 점검 명령과 green/skip 이유가 명확한 운영 health report.
ledger 스케일 opt-in rotation/archival + .idx 오프셋 인덱스까지 들어왔다 — rotate()가 segment로 봉인(체인은 seam 넘어 verify_chain() 검증)하고, .idx 사이드카로 event_count()/events_page()가 전체 파싱 없이 아카이브를 카운트·seek한다. 현 규모엔 read cache로 충분하므로 rotation/.idx는 대규모 운용 대비용. 정책(임계값)만 배포별로 조정.
문서 소스 정합성 이 HTML은 빠른 상태판이다. 상세 근거는 ADR, code review note, PR diff, ledger에 흩어져 있다. 상태판 생성 원천을 명확히 하거나, 최소한 갱신 날짜와 근거 commit을 항상 같이 갱신.
